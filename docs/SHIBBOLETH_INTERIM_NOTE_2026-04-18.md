# Shibboleth — where UCSD credentials come from, and what to do until then

*Date*: 2026-04-18
*Status*: Short procedural note; answers DK's 2026-04-17 question

---

## Short answer

For interim operation, the site uses ordinary password authentication: DK sets a password per user and the site validates it against `data/ka_auth.db`'s existing credentials surface. This is what the site runs on today.

For production (when we decide to invest the sprint), Shibboleth Single Sign-On against UCSD's Identity Provider requires (a) a Service Provider registration for `xrlab.ucsd.edu`, (b) an Apache/nginx Shibboleth SP module, (c) a metadata exchange with UCSD Identity Management, and (d) a production TLS certificate. The Shibboleth SP registration is the bureaucratic critical path and takes roughly two weeks.

## Where the credentials come from

UCSD's Shibboleth Identity Provider is operated by IT Services' Identity Management team. To register a Service Provider (the K-Atlas site), the path is:

1. **Request a Service Provider record** through UCSD ITS via the Identity and Access Management request form at https://support.ucsd.edu/its (search "Shibboleth SP" or "IdP integration"). The form asks for: the SP's public DNS name (`xrlab.ucsd.edu`), the entity ID (usually `https://xrlab.ucsd.edu/shibboleth`), the technical contact (DK or a designee), the set of attributes the SP needs released (typically `eduPersonPrincipalName`, `uid`, `mail`, `displayName`; for students also `eduPersonAffiliation`), and the data-protection assertion that the SP complies with UCSD's information-security policy.

2. **Receive and install the metadata.** UCSD ITS provides the IdP metadata URL and issues an SP metadata template. The SP metadata is generated by `shibboleth-sp` (Apache module) once installed.

3. **Install and configure the SP.** On the xrlab.ucsd.edu production VM: `apt install libapache2-mod-shib`, configure `/etc/shibboleth/shibboleth2.xml` with the entity ID and the session-handler path, point the Apache reverse-proxy at the Shibboleth-protected location, restart Apache. The existing `scripts/ka_sso_stub.py` FastAPI scaffold is what sits behind the Shibboleth-protected location; the SP module handles auth, the Python app reads `REMOTE_USER` / environment variables from the SP, and session state continues through the signed-cookie mechanism the stub already implements.

4. **Exchange metadata and test.** UCSD ITS adds the SP metadata to the campus federation; a test login from a UCSD SSO credential confirms attribute release. This is the step that routinely takes a week of back-and-forth.

5. **Request student-affiliation attribute release.** If the application needs to distinguish students from instructors (the K-Atlas role routing does), this requires an additional attribute-release agreement with ITS. Worth anticipating in the initial request.

The institutional-side contacts on campus are known to the UCSD Cognitive Science department's IT liaison; the Psychology and Cognitive Science ITS cluster has a designated Identity-Management liaison who handles most of these requests for the department's research sites. DK's department administrator can route the request.

## Why we are not doing this yet

Two reasons to defer.

1. **Time-to-value is poor for a small class.** The sprint to stand Shib up is 2–3 weeks of calendar time and ~ 10–15 hours of CW + ITS effort. With a class of 15–40 students and a password-authentication surface that already works, the bureaucratic investment is not justified for Spring 2026.

2. **Fall 2026 is a better inflection point.** If K-Atlas is still in active course use by Fall 2026, the Shibboleth sprint makes sense then — the cohort will be larger, the instructor will have more examples of student-onboarding friction, and the request-and-wait can run in the summer without blocking teaching.

## What the interim password auth actually is

The current state is:

- `data/ka_auth.db` has a `users` table with salted password hashes.
- `ka_login.html` and `ka_register.html` already implement registration and login against those hashes.
- DK can create an admin account and student accounts through `ka_admin.html` (Roster tab → add student) or, more reliably, by running an insertion script against `ka_auth.db` directly.
- Password-reset flows exist at `ka_forgot_password.html` but depend on an SMTP endpoint being wired; for now, password reset is done by DK issuing a new password manually and emailing it via the instructor's regular university account.

For Spring 2026, this is sufficient. Students sign up at the first class meeting with their UCSD email and a password DK hands them (or they set themselves at registration). The password is validated against the DB on every login. Sessions are signed cookies. There is no cross-site SSO, so students have to log in once per device.

When Shibboleth lands, the same `users` rows get `shibboleth_uid` populated and the password-login path remains available as a fallback for instructors and auditors who are not UCSD members.

## What I will not do on the interim path

- **I will not generate or rotate student passwords for DK.** Password generation and distribution is an instructor-in-person action. The site accepts passwords DK sets; it does not generate them in bulk.
- **I will not wire SMTP for password reset until we have real SMTP credentials.** The `ka_forgot_password.html` page is ready to call a `/api/auth/reset` endpoint; the endpoint is stubbed. We do not ship an SMTP integration without the sending-account credentials in place.
- **I will not bypass Shibboleth once it is live.** When Shib lands, the password-auth path stays available for non-UCSD users (auditors, external collaborators), but UCSD students authenticate through Shibboleth exclusively so the attribute-release (affiliation, email) is authoritative.
