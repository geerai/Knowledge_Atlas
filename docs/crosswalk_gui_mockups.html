Crosswalk GUI mockups — a design exploration

2026-04-18 · Working mockup over Codex's topic_crosswalk.json (102 defended rows × 18 outcomes × 9 architectural families). Four GUI patterns, same underlying data, rendered in-browser so the affordances can be felt before we commit to one. This page is a design artefact, not production UI.

What the crosswalk is and what it is not. Per docs/TOPIC_CROSSWALK_AND_TOPIC_PAGE_CONTRACT_2026-04-19.md, the crosswalk is a bridge layer between human-language outcomes (stress, attention, restoration, …), architectural families (luminous, acoustic, spatial, …), and defended Knowledge Atlas topic bundles. One row = one outcome × one architectural family × one topic bundle × its linked papers, theories, measures. It is not a rival taxonomy; it is the join surface that makes the taxonomies intelligible to a user who says "show me architecture and stress." The patterns below differ only in how they let a user travel across that surface.
1. Heatmap matrix (Tufte / Bertin)

The crosswalk's two root dimensions (architectural family × outcome) rendered as a 9 × 18 grid. Each cell's depth of colour encodes paper_count; empty cells are left grey. Clicking a cell drills into its defended bundles. The view shows at a glance where the evidence base is dense (luminous × cog.attention, natural × affect.restoration) and where it is thin.

paper count scale: 0 low mid high
Best for: honest overview — users see coverage density at a glance and cannot miss the gaps. Weak for: browsing when the user already knows their outcome but not the labels; requires visual scanning rather than search. Grounded in Bertin's Sémiologie graphique (1967) and Tufte's data-ink-maximisation principle (1983, 2006).
2. Faceted filter panel (Shneiderman dynamic queries)

Two chip rows — one for outcomes, one for architectural families. Clicking chips refines a visible bundle list in real time. Results update on every click; clearing chips returns to the full list. This is the direct-manipulation / dynamic-query pattern formalised by Shneiderman (1994) and used by every modern faceted-search UI from Amazon to Elicit.

Best for: the "show me X and Y" user who already has an intent. Discoverability is high because every chip shows its count. Weak for: users who do not know the vocabulary (the outcome chip labels can be opaque to a novice: what is affect.negative.stress?). Needs a plain-language relabel if this becomes the primary entry path. Grounded in Shneiderman's Dynamic Queries (1994) and Hearst's Search User Interfaces (2009).
3. Coordinated multi-view dashboard (Roberts CMV)

Three linked panels: outcomes (left), architectural families (middle), defended bundles (right). Selecting an outcome dims non-matching families and filters the bundle list; selecting a family does the reverse. The panels update in lockstep — a canonical Roberts-style multiple coordinated views system (2007). The user can enter from whichever side is most natural: a psychologist might click an outcome first, an architect an architectural family.

Best for: exploration when the user does not know which side to start from. The lockstep updates teach the structure of the bridge. Weak for: mobile screens (three columns cramped below 820 px); also has the highest visual-attention cost of the four patterns. Grounded in Roberts' State of the art: Coordinated and multiple views (2007) and the IVEE precursor (Ahlberg & Shneiderman, 1994).
4. Bidirectional lookup (plainest-possible)

Two dropdowns that mirror each other. Pick an outcome on the left, see the matching architectural families ranked by paper count on the left's output. Pick a family on the right, see the matching outcomes on the right's output. No matrix, no chips, no lockstep; just two one-way lookups side by side. The simplest pattern on offer.

Best for: novice users who want a single obvious thing. Highest usability on first encounter; every control labels itself. Weak for: users who want to see the whole landscape at once, or who want to filter by both axes simultaneously (this pattern does not compose filters). Grounded in Norman's Design of Everyday Things principle that affordances must be self-explanatory (1988, revised 2013).
5. Progressive-disclosure facets (Nielsen / Krug / Hearst)

The same faceted-filter idea as Pattern 2, but staged so the user never sees more than one decision at a time. The page opens with a single entry question ("where would you like to start?"), reveals only the first facet when the user commits to an entry path, groups its chips into plain-language categories shown as collapsed accordions, and unlocks the second facet and the bundle list only after the first selection. A breadcrumb lets the user back up; a small "show all facets" escape hatch jumps the power user to the full Pattern-2 surface. The underlying surface is the same crosswalk — what changes is the cognitive load at each moment.

Best for: first-time visitors, mobile users, and domains where the vocabulary is jargon to newcomers. Scaffolds the user through the decision rather than presenting everything at once. The breadcrumb teaches the taxonomy as a side effect. Weak for: power users who already know what they want — for them the extra click-through is friction; the "show all facets" escape hatch exists precisely to let them bypass the scaffold. Design lineage: Nielsen (1994) on minimalist design and progressive disclosure; Krug (2014) on "don't make me think"; Hearst (2009) §4 on faceted-search overwhelm-prevention; Wroblewski (2008) on form-design progressive reveal.

Comparison and recommendation

Pattern Best when Fails when Implementation cost
1. Heatmap matrix User needs an honest overview of where evidence is dense vs thin User cannot read the labels on the axes; does not afford search Low (~ 80 lines of HTML + JS, no libraries)
2. Faceted filter User has an intent ("architecture + stress") and wants to narrow Outcome labels are jargon; needs plain-language overrides Medium (~ 150 lines; responsive layout added complexity)
3. Coordinated multi-view User is exploring and wants to learn the taxonomy as they browse Mobile; high visual-attention cost; overkill for simple tasks High (~ 220 lines; lockstep state management; requires a real testing pass)
4. Bidirectional lookup First-time visitors who want one answer without exploring Users who want to combine filters or see the whole surface Very low (~ 50 lines, pure HTML select + one handler each)
5. Progressive-disclosure facets First-time visitors in a jargon-heavy domain; novices; mobile; overwhelm-prevention Power users (but they get a one-click escape to Pattern 2) Medium-high (~ 280 lines; staged state machine; accordion animations)
Updated recommendation (2026-04-18, after DK raised the overwhelm concern). Use Pattern 5 (progressive-disclosure facets) as the default landing view on ka_topics.html, with a prominent "Show all facets" link that jumps to Pattern 2 (full faceted filter) for users who know the vocabulary. Keep Pattern 1 (heatmap) as a secondary tab labelled Coverage map for audit-minded readers. Rationale: Pattern 5 scaffolds the user through the decision without hiding the option to see everything at once. The first screen is a single three-way question that orients the user to the bridge's two axes; the accordion grouping of outcomes into four plain-language categories (Cognitive, Affective, Physiological & Health, Behavioural, Other) relabels the taxonomy in terms a non-specialist can read; the orthogonal-facet reveal shows only options with defended bundles, so the user never clicks into a cell that's in the evidence gap; and the breadcrumb both confirms the user's path and teaches the structure. Pattern 2 remains available one click away for the power-user case, which is the right trade-off per Krug's "don't make me think" principle — default to the path that never makes the user think, escape-hatch the advanced view for users who already have. Pattern 3 stays deferred; Pattern 4 is suitable for a landing-page widget but not for the full topic page.
