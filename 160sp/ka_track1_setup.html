Track 1 Setup — Image Tagger

Track 1 uses two required repositories and one optional companion. The Image Tagger produces the per-image latent-variable tags; the Tagging Contractor manages the contracts (schemas, acceptance criteria, validation sweeps) that tell the Tagger what a legitimate tag set looks like. An optional third repository, the Outcome Contractor, layers in outcome-defined search: it lets a user query the tagged image base for photographs that are predicted to produce a particular psychological outcome (say, restoration, or threat-perception). Plan for about forty-five minutes end-to-end, longer if this is your first time installing Python or Git.

Prerequisites

If you have already completed the site-wide install in Technical Setup, you can skip straight to Step 3. Otherwise, work through the prerequisites below before touching the Tagger itself.

  • Python 3.11 or later — confirm with python3 --version.
  • Git — confirm with git --version.
  • A GitHub account with SSH keys configured (one-time; see the course tech-setup page).
  • 8 GB free disk for the sample image corpus and the cached vision models.
macOS ships with an older Python. Install a modern one via Homebrew: brew install python@3.12. Windows users should install Python from python.org and tick "Add Python to PATH" in the installer, or use winget install Python.Python.3.12.

Step-by-Step

1Clone the two required repositories

Create a parent folder (for instance ~/REPOS/) and clone both repositories side by side:

mkdir -p ~/REPOS && cd ~/REPOS

# Required repo A — the tagger itself
git clone git@github.com:kirsh-lab/image_tagger.git

# Required repo B — the contracts that govern what a legitimate tag set is
git clone git@github.com:kirsh-lab/Tagging_Contractor.git
Why two repositories? The image_tagger holds the vision models, rule-based extractors, and the CLI that turns a photograph into a JSON tag set. The Tagging_Contractor holds the contracts — the JSON-schema definitions, acceptance criteria, golden examples, and validation sweeps that decide whether a tag set is admissible. The Tagger produces; the Contractor adjudicates. Keeping them separate lets methodologists revise the contracts without touching the model code and lets engineers revise the model without renegotiating the contracts.
2(Optional) Clone the Outcome Contractor
# Optional third repo — only if your sub-team needs outcome-defined image search
git clone git@github.com:kirsh-lab/Outcome_Contractor.git

Clone this third repository only if your sub-team is working on outcome-based image retrieval. The Outcome Contractor defines a second class of contract: rather than adjudicating whether a tag set is admissible, it defines what it means for a photograph to produce a given outcome — restoration, threat, social ease, wayfinding clarity. Once populated, it lets a user query the tagged image base with requests such as "show me images predicted to elicit restoration for an office-worker user profile." Most students will not touch this repository in Weeks 3–5; it is surfaced here so that those who will can clone it alongside the other two from the start.

3Create and activate a virtual environment

Create one shared environment at the parent level so both repositories import against the same dependency set:

# macOS / Linux
cd ~/REPOS
python3 -m venv .venv
source .venv/bin/activate

# Windows (PowerShell)
cd $HOME\REPOS
py -3 -m venv .venv
.venv\Scripts\Activate.ps1
4Install dependencies for both repositories
pip install --upgrade pip

# Tagger — vision models and rule-based extractors
pip install -r image_tagger/requirements.txt

# Contractor — JSON-schema tooling and validation sweeps
pip install -r Tagging_Contractor/requirements.txt

# Outcome Contractor (only if cloned)
pip install -r Outcome_Contractor/requirements.txt

First install pulls OpenCV, PyTorch, and the COCO-133 segmentation weights. Expect 5–10 minutes on a reasonable connection.

5Download the sample image corpus
python image_tagger/scripts/fetch_sample_images.py --set cogs160_week3

Roughly 120 photographs (living rooms, offices, corridors, outdoor transitional spaces) pre-licensed for classroom use.

6Run the tagger on one image to confirm install
python -m image_tagger.cli tag \
    --image image_tagger/data/samples/cogs160_week3/room_014.jpg \
    --output out/room_014.tags.json

Open out/room_014.tags.json and you should see object-material detections, abstract visual statistics (fractal dimension, visual complexity, openness), and any L-code latent variables the rule-based extractors currently emit. If the JSON looks empty, the model download failed — re-run Step 4.

7Validate the tag set against the contract
python -m tagging_contractor.validate \
    --tags out/room_014.tags.json \
    --contract Tagging_Contractor/contracts/cogs160_week3.contract.json

This is the round trip that makes Track 1 work: the Tagger produces a JSON object and the Contractor either admits it (all schema, coverage, and consistency checks pass) or returns a list of violations that tell you exactly what the Tagger got wrong. Your weekly contributions will usually be either (a) a new rule-based extractor in the Tagger or (b) a tightened contract clause in the Contractor — rarely both in the same PR.

8Boot the local site (optional but recommended)
python Knowledge_Atlas/160sp/ka_local_server_setup.py  # serves at http://localhost:8765

This lets you browse the class site exactly as it is deployed, including the Track 1 page you came from. (If you have not cloned Knowledge_Atlas yet, do so now — git clone git@github.com:kirsh-lab/Knowledge_Atlas.git.)

Common Snags

  • SSL errors on pip install — usually a stale pip. Rerun pip install --upgrade pip certifi.
  • Torch install hangs on Windows — swap to the CPU wheel: pip install torch --index-url https://download.pytorch.org/whl/cpu.
  • "image_tagger not a package" — you ran the CLI from the wrong directory. cd back to ~/REPOS (the parent that contains both repositories) before re-running.
  • Contractor says "contract file not found" — the --contract path is relative to your current working directory. Use an absolute path or run the validator from the ~/REPOS parent.
Branching rule. Never commit directly to main. Create a branch named track1/<your-github-handle>/<short-topic> for every change, open a PR, and request review from your track lead.
