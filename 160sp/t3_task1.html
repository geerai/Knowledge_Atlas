TRACK 3 · TASK 1

Infinigen Setup, Coverage Survey, and Parameter Manifests

For each room type you are assigned, you will ship a JSON-Schema parameter manifest — a typed, range-constrained, citation-backed list of the five to seven knobs an LLM front-end is allowed to turn. You do not segment meshes. You do not author 3D files. You design the parameter contract that downstream tasks honour.

5 phases 75 points Weeks 3–5

The big picture

Track 3 builds the parametric backbone of the Knowledge Atlas’s VR pipeline on top of Infinigen Indoors (Raistrick et al., 2024), the Princeton VLL procedural-generation system released under the BSD licence. Students never touch Blender’s graphical interface. A parametric room, in this codebase, is a Python entity whose geometry, materials, and arrangement regenerate from a small set of named, typed, range-constrained parameters; turning the knobs sweeps the room through a controlled space of variations. An LLM front-end (your Task 3 deliverable) will read the parameter schema and write valid patches in response to natural-language prompts such as “raise the ceiling and warm the lighting.”

The single most important artefact a Track 3 student ships is therefore the parameter manifest: a JSON Schema document specifying which five to seven parameters per room type are exposed, what each parameter means, what units and ranges it accepts, and what published environmental-psychology evidence motivates exposing it at all. The manifest is, in Cronbach & Meehl’s (1955) sense, a construct-validity document. Each parameter you expose is a claim that this knob corresponds to a measurable construct in the literature; each range you set is a claim about the band over which that construct has been studied. When Vartanian et al.’s (2015) cathedral-effect work finds that ceilings between 2.4 m and 3.0 m modulate abstract-thinking measures, your ceiling_height_m parameter inherits both the construct and the band.

Infinigen ships with seven indoor room types out of the box: kitchen, living room, bedroom, bathroom, dining room, hallway, and office. Eight types from the cohort’s fifteen are not yet covered: classroom, hospital, restaurant, lobby, library, retail, museum, gym, and worship. For un-covered types you will draft an extension proposal — a parameter manifest that specifies how the closest existing Infinigen entity should be adapted — as if it will eventually be implemented. Worship is reserved for the cohort’s flagship Task 3 deliverable and is excluded from Task 1.

What “parametric” means here

In this track, “parametric” carries the strong sense used in procedural geometry: geometry, materials, and arrangement are generated by mathematical rules whose parameters can be swept (Raistrick et al., 2024). It is not the looser sense in which a CAD-authored scene with movable furniture is sometimes called parametric. The theoretical lineage runs through Stiny’s (1980) shape grammars: a room is the output of a generative procedure, and the parameters are the controllable inputs to that procedure. A parameter you expose must therefore correspond to a real knob in the generator’s code — not a label on a finished mesh.

Phase 1 — Install Infinigen Indoors

Required tooling

  • Blender 4.x (CLI access only; no GUI work)
  • Python 3.10+ with a clean virtual environment
  • Infinigen Indoors from princeton-vl/infinigen (BSD licence)
  • Course-supplied wrapper: Knowledge_Atlas/scripts/infinigen_wrapper.py
  • git, jq, and jsonschema (for manifest validation)

Install sequence

# 1. Create a clean Python environment
python3 -m venv ~/venvs/infinigen
source ~/venvs/infinigen/bin/activate
pip3 install --upgrade pip

# 2. Clone Infinigen Indoors and install in editable mode
git clone https://github.com/princeton-vl/infinigen.git ~/code/infinigen
cd ~/code/infinigen
pip3 install -e ".[indoors]"

# 3. Install the manifest validator the course uses
pip3 install jsonschema

# 4. Smoke test: generate a default living room headlessly
python3 -m infinigen_examples.generate_indoors \
  --task indoor_scene_trial \
  --output_folder ~/infinigen_out/smoke_test \
  --seed 0 \
  --configs singleroom.gin fast_solve.gin \
  --pipeline_configs local_64GB.gin

# 5. Confirm output
ls ~/infinigen_out/smoke_test
cat ~/infinigen_out/smoke_test/scene.json | head -40

Phase 2 — Coverage survey

For each of Infinigen’s seven built-in indoor room types, run the generator twice — once at default parameters and once at one randomised seed — and document the entity hierarchy. Each Infinigen room is a tree of named entities (Wall, CeilingLight, WindowFrame, Couch, and so on). Your job is to list the top-level entities and the parameters they expose, so that Phase 3 can pick a defensible subset.

Phase 3 — Author parameter manifests for two assigned rooms

The instructor will assign each student two of the seven built-in room types. For each, write a JSON-Schema parameter manifest exposing five to seven parameters. Each parameter must declare its name, type, units, range, default, one-line description, and at least one published environmental-psychology citation. At least one parameter per manifest must cite a specific empirical study from the references list (Vartanian et al., 2015; Kaplan & Kaplan, 1989; Ulrich, 1991; Münch et al., 2020; Meyers-Levy & Zhu, 2007; Joye, 2007).

Ask your AI

I’m authoring a JSON-Schema parameter manifest for {room_type}. The manifest exposes 5–7 parameters that an LLM front-end can change. For each candidate parameter, tell me: (a) is it actually exposed by Infinigen’s {room_type} entity in the current source, (b) what is the strongest published environmental-psychology study that motivates including it, with full citation, and (c) what range is defensible given the bands actually studied? Push back hard if any candidate is poorly motivated, redundant with another parameter, or outside the empirical range.

Deliverable

Two manifests in track3/your-name/manifests/ — {room_type_a}.manifest.json and {room_type_b}.manifest.json — plus a one-page rationale document track3/your-name/manifests/RATIONALE.md citing every study in APA form and explaining each range choice.

Phase 4 — Extension proposals for un-covered room types

Pick at least one room type from the eight Infinigen does not cover (classroom, hospital, restaurant, lobby, library, retail, museum, gym). For your chosen type, draft an extension proposal: identify the closest existing Infinigen entity, name five to seven additional parameters that would adapt that entity to the new typology, and justify each with published literature. Treat the proposal as a specification for an engineer who will eventually implement the new entity.

Required structure of an extension proposal

  • Target room type and the closest existing Infinigen entity
  • Gap analysis: what the existing entity cannot express that the new typology requires
  • Proposed new parameters in the same JSON-Schema form as Phase 3, marked "status": "proposed"
  • Implementation hooks: which infinigen source files would need editing (file paths, not patches)
  • Literature backing for each parameter
Worship is excluded from Phase 4. Worship spaces are the cohort’s flagship Task 3 deliverable and are handled there with full instructor scaffolding. Do not propose extensions for worship in this task.
Deliverable

track3/your-name/extension_proposals/{room_type}.proposal.md containing the five sections above plus the proposed JSON-Schema fragment.

Phase 5 — Peer review

Each student peer-reviews two other students’ manifests. The review is a construct-validity audit, not a typography check. For each manifest you review, file at least three concrete issues, in track3/your-name/peer_review_<reviewee>.md.

Your review must check, for every parameter: (a) does the parameter cite a real study, (b) does the cited study actually support exposing this parameter (not just mention the construct in passing), (c) are the units and ranges defensible against the bands the literature reports, and (d) does the parameter map cleanly to a JSON-Schema field that an LLM could safely write to without producing illegal Infinigen state? Treat “safely” literally: an out-of-range write must fail validation rather than crash the generator.

Deliverable

Two peer-review files committed to your branch, each with at least three concrete construct-validity issues raised.
